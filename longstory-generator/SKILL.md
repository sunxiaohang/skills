# LongStory-Generator：长篇小说结构化生成器

**风格定位：搞笑无厘头 + 逻辑自洽**

本 Skill 将长篇小说创作拆解为三个强制确认阶段，每阶段均有内部自检机制，确保风格一致、逻辑自洽、无遗漏。

---

## 输入格式要求

用户需提供（若未提供则主动询问）：

- **小说背景**：世界观、时代、核心冲突、风格基调（玄幻/悬疑/都市等）。
- **创作方向**：希望表达的主题、主角初步设想、预期篇幅或节奏（快节奏网文 / 慢热文学）。

收到用户输入后，直接进入第一步，无需额外确认。

---

## 第一步：生成故事创意梗概与核心框架

### 1.1 输出内容（总字数 ≤ 2000 字）

#### A. 故事创意梗概（300 字内）

- **一句话核心**：主角 + 核心冲突 + 独特无厘头亮点。
- **扩展段落**：世界观亮点、主角身份、核心冲突、结局暗示。
- **风格基调**：明确标注"搞笑无厘头，逻辑自洽型"，并给出 1-2 句风格示例。
  > 示例：主角是一个想当大侠的乞丐，但他唯一的武功是用脚趾头夹飞镖——不巧，这个世界的武功强弱取决于脚气浓度。

#### B. 核心框架（1500 字内）

- **三幕或起承转合结构**：每幕的核心功能、主要情节点、情绪曲线。
- **章节级大纲**：至少 10 个章节，每章包含：
  - 章节标题（暂定）
  - 核心事件（1-2 句话）
  - 结尾钩子类型（悬念 / 反转 / 情绪 / 任务 / 无厘头式意外）
- **关键冲突与转折点**：列出前 3 个主要转折点及对应章节位置。
- **主题与动机**：主角的核心欲望与核心恐惧，故事想探讨的母题。

### 1.2 内部自检（自动执行，不通过则拒绝进入下一步）

- [ ] 主角有明确的欲望和恐惧。
- [ ] 核心冲突具有可持续性（能支撑长篇）。
- [ ] 每个章节都有功能性（推进剧情或深化人物）。
- [ ] 钩子分布均匀，至少每 3 章有一个强钩子。
- [ ] 风格基调中至少包含一个"荒诞但可自圆"的示例。

### 1.3 用户确认

输出完成后询问：

> "请确认以上创意与框架。如需微调（如修改某章内容、改变结局方向），请指示。确认后进入第二步。"

---

## 第二步：生成关键上下文文档（故事圣经，约 1 万字）

### 2.1 输出内容结构（严格按以下模块，总字数 ≈ 1 万字）

#### 【模块 A】基础设定卡（约 1000 字）

- 世界观名称、核心法则（魔法、科技、社会制度等）。
- 时代背景（年份、纪年、历史大事件）。
- 地理与势力分布（地图简述、主要阵营）。
- 特殊名词/术语表（附解释）。
- 该世界的"荒诞规则示例"（至少一条无厘头但一致的设定）。

#### 【模块 B】人物档案（约 3000 字）

每个主要人物独立卡片，包含：

| 字段 | 说明 |
|------|------|
| 姓名、绰号、年龄、性别 | 基本信息 |
| 外貌特征 | 标志性穿着、疤痕、眼神等 |
| 性格关键词 | 3-5 个，附带具体行为示例 |
| 核心欲望 | "我想要……" |
| 核心恐惧 | "我害怕……" |
| 口头禅/习惯动作 | 可含无厘头特色 |
| 能力/技能等级 | 若适用 |
| 角色弧光 | 起始状态 → 变化 → 最终状态 |
| **搞笑属性** | 该人物的主要笑点来源 |

次要人物简表：姓名 + 身份 + 一句话作用。

#### 【模块 C】人物关系图谱（约 1500 字）

- 情感关系（爱情、友情、敌对、师徒、利益绑定等）
- 权力关系（上下级、制衡、背叛可能性）
- 历史纠葛（过去的恩怨、误会）
- 关键关系的事件锚点（例如"A 曾救过 B → B 对 A 绝对忠诚"）
- **无厘头关系点**（例如"C 和 D 是死敌，但因为都害怕主角的脚气而临时结盟"）

#### 【模块 D】故事详细脉络（约 3000 字）

按三幕/起承转合分段，每段内列出核心情节点、主角阻碍、配角高光时刻。

**分卷/分章细纲**：至少 20 个章节，每章包含：

| 字段 | 说明 |
|------|------|
| 章节编号与暂定标题 | — |
| 场景概要 | 何时、何地、何人、何事 |
| 本章目标 | 推动主线 / 揭示信息 / 人物成长等 |
| 结尾钩子 | 具体写法示例，可含无厘头型 |
| **风格提醒** | 若本章有特别离谱的笑点，提前标注其逻辑自圆方式 |

#### 【模块 E】框架结构拆解（约 800 字）

- 情绪节奏图（高潮与低谷分布）。
- **伏笔与呼应清单**（表格形式）：

  | 伏笔 ID | 埋设章节 | 回收章节 | 内容描述 | 笑点逻辑锚点（若有） |
  |---------|---------|---------|---------|-----------------|

- 时间线表格：关键日期/天数，标注事件。

#### 【模块 F】风格与一致性指南（约 700 字）

- 叙事视角（第一人称 / 第三人称有限 / 全知）。
- 语气基调：搞笑、轻松、明快，带吐槽感。
- 禁止使用的词汇或表述（避免现代用语穿越到古代等）。
- 专有名词拼写统一规则。

**F-4. 搞笑无厘头风格细则（强制）**

- 核心原则：荒诞但不失控，笑点服务于人物或情节，内部逻辑自洽。
- 具体手法（写作时至少选用 2-3 种）：
  1. **反逻辑但自圆**：看似离谱的行为，后续揭示其有一套歪理或特殊世界观设定。
  2. **夸张与反差**：严肃场景中突然插入荒谬细节，并立即给出合理原因。
  3. **吐槽与冷幽默**：角色对荒诞事件做理性分析，制造反差。
  4. **无厘头对话**：跳脱常规，答非所问但暗含逻辑。
  5. **意外翻转让严肃崩塌**：精心计划因极小概率的愚蠢原因失败。
- 禁止事项：为搞笑破坏核心设定、低俗笑点、笑点密度过高（每千字 2-4 个为佳）。

**F-5. "经得住推敲"的保障机制**

- 每章自检问题：
  - 每个"不合理"行为，在小说世界观内是否有逻辑链条？
  - 是否至少有一个笑点同时推进了情节或揭示了人物性格？
  - 有没有笑点与前面章节的设定矛盾？
- **笑点记录表**：记录每个笑点的触发方式、逻辑自圆句、所在章节。

### 2.2 内部自检（自动执行）

- [ ] 每个主要人物都有完整的角色弧光。
- [ ] 人物关系至少有一个潜在冲突点。
- [ ] 伏笔清单中每个伏笔都标注了预期回收章节。
- [ ] 时间线无逻辑漏洞。
- [ ] 整份文档字数误差 ≤ 10%（9000-11000 字）。
- [ ] F-4 和 F-5 内容完整且可执行。

### 2.3 用户确认

输出完成后询问：

> "关键上下文文档已完成（约 1 万字）。请确认是否满足需要，或要求补充某模块（如增加配角、调整伏笔）。确认后进入第三步。"

---

## 第三步：生成小说正文

### 3.1 写作策略

- **分章生成**：每次只生成 1 章（约 2000-4000 字）。
- **前置记忆**：每次生成前自动注入：
  - 故事圣经核心要素（人物卡 + 关系 + 伏笔清单 + 风格细则）
  - 已生成章节的摘要列表（最近 3 章详细摘要）
  - 当前章节的大纲（来自第二步细纲）

**指令模板（每章自动填充）：**

```
基于以下故事圣经：[插入第二步文档的关键摘要]
前情提要：[最近一章结尾的摘要]
请严格按照细纲生成第 X 章《章节标题》。

要求：
1. 主角 [名字] 的外貌/习惯需保持一致（如：他总是不自觉摸耳垂）。
2. 本章结尾必须使用钩子：[钩子类型 + 具体示例]。
3. 对话要体现 [人物 A] 的口头禅："XXXX"。
4. 不得提前揭露尚未到回收章节的伏笔（伏笔 ID：F-003 计划在第 8 章回收）。
5. 全文采用搞笑无厘头但逻辑自洽的风格：
   - 每段正文中至少包含一处荒诞情节或对话，荒诞发生后必须在本章内给出符合世界观的解释。
   - 主角或配角可担任"吐槽役"，对荒诞事件进行理性分析，制造反差。
   - 禁止低俗笑点；所有笑点需能回溯到人物性格或世界设定。
6. 在章节末尾，用一句"逻辑自圆检查"说明本章最离谱的情节如何自洽。
7. 风格一致性：保持全篇语气轻松、节奏明快。
```

### 3.2 每章生成后的自检

- [ ] 主角名称、外貌、能力与人物卡一致。
- [ ] 所有专有名词拼写正确。
- [ ] 本章事件与时间线无冲突。
- [ ] 结尾钩子明确，且与细纲一致。
- [ ] 未出现设定文档之外的"即兴设定"（如有，提示用户决定是否保留）。
- [ ] 符合 F-4 的搞笑无厘头细则。
- [ ] 已输出"逻辑自圆检查"语句。

### 3.3 阶段性拼接与平滑

- **每 5 章**：自动执行一次一致性检查，输出报告：
  - 人物状态变化
  - 伏笔埋设与回收进度
  - 时间线核对
  - 笑点密度统计

- **全文完成后**：执行最终平滑：
  - 合并所有章节为单个文本。
  - 检查章节间过渡句是否自然，为缺失处自动生成过渡段落。
  - 统一叙事视角与语气。
  - 生成**伏笔回收对照表**（埋设章节 → 实际回收章节 → 是否呼应）。
  - 生成**人物状态追踪表**（从第一章到最后的变化）。
  - 生成**笑点密度与逻辑自圆统计表**（每章笑点数量、最离谱情节及其解释章节）。
  - 检查全篇是否有违背"经得住推敲"原则的笑点，如有则自动提示建议修改。

### 3.4 输出格式

- 每章单独输出，标题为"第 X 章 [章节名]"。
- 最终提供一个完整文本文件（或可复制的整合区）。

---

## 无遗漏保障机制

- 每一步都有强制检查清单，内部自动验证，若检查不通过则拒绝进入下一步并给出修正建议。
- 第二步的上下文文档包含风格与一致性指南，写作时持续引用。
- 每章生成前重新加载最近 3 章的摘要，避免跨章矛盾。
- 最终整体做两次扫描：逻辑一致性扫描 + 风格一致性扫描。
