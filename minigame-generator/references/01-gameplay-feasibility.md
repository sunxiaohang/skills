# 玩法可行性评估框架

## 一、立项前必答 10 问

在写一行代码之前，逐项回答以下问题。任何一项为「否」则需重新设计：

| # | 问题 | 通过标准 |
|---|------|----------|
| 1 | **30 秒理解规则** — 玩家能否在 30 秒内理解核心操作？ | 核心操作 ≤ 3 步 |
| 2 | **碎片化友好** — 单局时长是否在 2-5 分钟？ | 最短局 ≥ 1 分钟，最长局 ≤ 10 分钟 |
| 3 | **Canvas 2D 可行** — 所需视觉效果是否能用 Canvas 2D 实现？ | 无 3D、无骨骼动画、无复杂粒子物理 |
| 4 | **横屏触控适配** — 操作是否适合手机横屏（≤ 2 根手指）？ | 单手/双手握持均可操作 |
| 5 | **即时反馈** — 每个操作是否有 ≤ 100ms 的视觉+音效反馈？ | 无「按下无反应」的操作 |
| 6 | **重玩动力** — 是否有随机性或渐进解锁驱动重玩？ | 至少 1 个重玩驱动系统 |
| 7 | **包体可控** — 纯代码+Emoji 绘制，无大型资源？ | 主包 ≤ 4MB |
| 8 | **性能达标** — 同屏最大实体数是否在 Canvas 2D 性能范围内？ | 同屏实体 ≤ 500 |
| 9 | **社交属性** — 是否有分享/排行榜/对比等社交切入点？ | 至少 1 个社交功能 |
| 10 | **差异化** — 与微信小游戏平台已有同类游戏有何差异？ | 至少 1 个核心差异点 |

## 二、游戏类型适配矩阵

不同游戏类型在 Canvas 2D 小游戏平台上的适配度评估：

### 高适配（推荐）
| 类型 | 代表作 | 适配理由 |
|------|--------|----------|
| 双摇杆射击/动作 | AssassinLines | 操作直觉、视觉冲击力强、纯几何绘制 |
| 六边形/棋盘策略 | ClashKing | 回合制/半实时、低帧率需求、策略深度 |
| 弹幕躲避 | — | 大量小型实体、Canvas 2D 擅长批量圆形绘制 |
| 塔防 | — | 固定视角、实体有限、策略+数值驱动 |
| 消除/三消 | — | 棋盘式布局、动画简单、成瘾性强 |
| 合成/合并 | — | 低操作频率、收集驱动、长线留存 |

### 中适配（需验证）
| 类型 | 风险点 |
|------|--------|
| 平台跳跃 | 精密操作在触屏上体验差 |
| 卡牌对战 | UI 信息量大、文字多、适合竖屏 |
| 模拟经营 | 系统复杂度高、开发周期长 |

### 低适配（不推荐）
| 类型 | 原因 |
|------|------|
| 3D 游戏 | Canvas 2D 无法实现 |
| 格斗/动作RPG | 操作复杂度超出触屏承载 |
| 大世界开放 | 资源量和复杂度远超小游戏限制 |
| 实时多人 PVP | 网络延迟和同步问题 |

## 三、核心循环设计模板

每个小游戏必须有清晰的核心循环，遵循以下模板：

```
┌─────────────┐
│   输入       │  ← 玩家操作（触摸/滑动/点击）
│  (Input)     │
└──────┬──────┘
       ↓
┌─────────────┐
│   处理       │  ← 游戏逻辑（碰撞/AI/数值计算）
│ (Process)    │
└──────┬──────┘
       ↓
┌─────────────┐
│   反馈       │  ← 视觉 + 音效 + 数值变化
│ (Feedback)   │
└──────┬──────┘
       ↓
┌─────────────┐
│   奖励       │  ← 分数/解锁/成就/排行变动
│ (Reward)     │
└──────┬──────┘
       ↓
       └──→ 回到「输入」（循环）
```

### 循环节奏参考

| 循环类型 | 节奏 | 适用游戏 |
|----------|------|----------|
| 超快循环 | 0.5-2 秒/次 | 弹幕、节奏、消除 |
| 快循环 | 2-5 秒/次 | 动作、射击（AssassinLines） |
| 中循环 | 5-30 秒/次 | 策略、塔防（ClashKing） |
| 慢循环 | 30-120 秒/次 | 经营、养成 |

## 四、操作手势可行性

### 已验证可靠的手势

| 手势 | 实现难度 | 适用场景 | 实例 |
|------|----------|----------|------|
| 双摇杆 | ★★★ | 移动+攻击 | AssassinLines |
| 单指拖拽 | ★☆☆ | 地图平移、棋子移动 | ClashKing 镜头 |
| 单指点击 | ★☆☆ | 选择、确认 | 菜单、选关 |
| 双指缩放 | ★★☆ | 地图缩放 | ClashKing 镜头 |
| 长按蓄力 | ★★☆ | 蓄力攻击 | AssassinLines 瞬移 |
| 滑动翻页 | ★★☆ | 菜单切换 | 选关界面 |

### 应避免的手势

| 手势 | 原因 |
|------|------|
| 三指及以上 | 手机屏幕空间不足 |
| 精确拖拽到特定位置 | 手指遮挡目标 |
| 快速双击（≤ 200ms） | 误触率高 |
| 边缘滑动 | 与系统手势冲突 |

## 五、性能预算

在立项时估算性能预算，确保 Canvas 2D 60fps 可达：

| 资源 | 预算上限 | 说明 |
|------|----------|------|
| 同屏绘制调用 | ≤ 800 draw calls/帧 | 含粒子、UI、游戏对象 |
| 粒子数量 | ≤ 600 | 布朗粒子 + 爆炸粒子总和 |
| 游戏对象 | ≤ 200 | 敌人+子弹+道具+障碍物 |
| UI 文本渲染 | ≤ 30 次/帧 | `fillText` / `strokeText` |
| Canvas 状态切换 | 尽量少 | `save()/restore()` 配对 |
| 每帧内存分配 | ≈ 0 | 热路径禁止 new/[] |

## 六、最小可玩版本（MVP）定义

每个新游戏的 MVP 必须包含：

- [ ] 核心玩法循环可运行
- [ ] 至少 1 种敌人/挑战
- [ ] 胜负判定正常
- [ ] 基础 HUD（生命/分数/进度）
- [ ] 菜单（开始/重试）
- [ ] 触控输入正常（含安全区域）
- [ ] 横屏 + DPI 适配
- [ ] 至少 1 个音效
- [ ] 布朗粒子背景
- [ ] 帧率稳定 ≥ 50fps
